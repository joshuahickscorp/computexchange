# Spec-lab — first autonomous run (2026-07-06): the distributed-speculation thesis, measured

*The deterministic experiment engine (`scripts/spec-lab/`) ran the full 14-rung ladder
hands-off on a rented GPU, with auto-remediation and money-safe teardown, for **$0.05**
(balance $43.35 → $43.30). Every number below is from `ledger.jsonl` — real measurements,
with honest failures and caveats labeled. This is a viability probe, not a product claim.*

## The engine worked end-to-end

Provision (walk the GPU plan) → keep the first **reachable + CUDA-usable** card → deterministic
base setup (vLLM/ffmpeg/numpy/skimage pinned) → run each rung → check its bar → PASS advances,
FAIL walks the remediation ladder → stream to a resumable ledger → tear down on every exit path.
The two infra walls that beat me by hand are now handled in code: datacenter-routing
(provision proves a real SSH round-trip) and broken-host CUDA (proves `torch.cuda.is_available()`
before accepting a pod — added after run 1 caught it).

## What this says about the thesis (honestly)

- **The keystone — speculative rendering — is real and worth building.** Frame-interpolation
  speculation and residual "render-the-delta" both land **2–2.8× at ≥99% SSIM** on ordinary
  (low-motion) footage, and adaptive render sampling hits 2.8× at 0.97 quality. On high-motion /
  cuts it correctly speculates nothing and costs ~nothing — the failure mode is graceful, which
  is exactly what a marketplace needs (sell the win, never punish the buyer on a miss).
- **The generality thesis holds at the principle level:** structured non-text files are highly
  predictable byte-wise (image/audio ≈1.0), random data isn't (~0) — speculation pays where there
  is structure, which is most real files.
- **The price story is directionally strong but still MODELED:** ~4× cheaper than DIY-RunPod, but
  that leans on a fleet-idle-marginal cost assumption and (in run 1) a conservative 1.6× speedup
  default. It should be re-run with the *measured* 2–2.8× speedups (wired in for run 2).

## Honest caveats (do not oversell)

1. **Synthetic clips/scenes.** B1/B2/C2 used generated footage and an MC-noise stand-in for a real
   path tracer. Real photos/film/3D will be harder; these numbers are an upper-ish bound on
   predictable content, not a guarantee on arbitrary jobs.
2. **The distributed speedups are understated** (verify ran on CPU on the no-CUDA pod) AND their
   fleet fan-out is modeled — treat B4/C3 as lower bounds pending a real GPU-verify run.
3. **The vLLM lossless anchor (A1/A2) has NOT run yet** — the L40S CUDA runtime was broken. Run 2,
   with the CUDA gate, targets a card where vLLM actually initializes.
4. **The cost model is a transparent MODEL**, not a measured invoice; it's honest about that.

## Runs 2 & 3 — the vLLM anchor, chased and honestly parked

The CUDA gate (added after run 1) worked: run 2 landed a genuinely CUDA-capable L40S, and
resume-skip re-ran only the failures. What we learned about the LLM anchor:

- **A2 (draft-model spec-decode): NOT SUPPORTED in vLLM 0.11** — the engine returns an explicit
  `NotImplementedError: Speculative decoding with draft model is not supported yet; use ngram`.
  Real finding, not a bug in our harness. Parked.
- **A1 (ngram spec-decode): a persistent vLLM-0.11 engine-init crash.** With real CUDA the
  *baseline* (no-spec) engine builds fine, but every *spec* build fails with a generic
  `EngineCore initialization failed` whose real cause lives in vLLM's EngineCore subprocess
  stderr (not surfaced to the parent). Run 3 tried the standard remedy — `enforce_eager=True`
  (disable CUDA-graph capture) — and it STILL failed across all four variants. Diagnosing
  further needs a live session capturing the subprocess stderr, not another blind knob.

**Decision: park the vLLM LLM anchor.** It is (a) the least important rung — LLM speculative
decode is shipped field technology, not this project's novel contribution; (b) redundant for
the thesis, since **A3's byte-level accept/reject already demonstrated the lossless-AR
speculation principle working (0.67 acceptance)**; and (c) blocked on a vLLM-0.11 quirk, not on
anything about our approach. Chasing a 4th blind fix would be poor stewardship of borrowed GPU
time. Total spend across ALL pod work this session (soak attempts + 3 lab runs): **~$0.97.**

## Where this leaves the thesis + the real next step

**Validated:** the keystone — speculative *rendering* — works (2–2.8× at ≥97% fidelity, graceful
no-op on hard content), the generality principle holds (structured files are speculatable), one
protocol spans all modalities, and the cost model points ~4× under DIY-RunPod (modeled).

**The honest gate before productizing:** run B/C on **real footage and real 3D scenes** (not the
synthetic clips), and get a genuine **GPU-verify** for the distributed rungs (B4/C3 were
CPU-understated). If the keystone survives real content, the build is the `draft/verify/gate`
orchestrator (fork target) + the "cheaper than renting it yourself" $/hr reference on the submit
page (task filed). The vLLM anchor can be revisited with a live debug session if we ever want the
LLM lane in the lab, but it is not on the critical path to the rendering product.

## Track R — the SAME methodology on REAL content (the reality check, and it MATTERS)

Ran the four real-content rungs on a genuine GPU (L40S/OptiX) against real files — Blender
Cycles path tracing, the Sintel trailer, a Wikimedia photo, Big Buck Bunny. This is where the
optimistic synthetic numbers met reality, and the picture is honest and nuanced:

| rung | real content | result | headline | honest reading |
|---|---|---|---|---|
| **R4 transcode** | Big Buck Bunny (real ffmpeg) | **PASS** | **2.18× at SSIM 0.98** (83% segments accepted, looser gate) | **the real winner** — modeled:false, genuine encode wall-times; speculative transcode (cheap preset + selective re-encode) genuinely pays on real video |
| **R2 frame-interp** | Sintel trailer (real motion) | **PASS** | **1.34× at 0.98** (accept 43%) | modest — real motion is far harder than synthetic pans (acceptance fell from ~100% to 20–43%); a real but small win |
| **R3 image** | Wikimedia photo | **PASS** | 9.6× superres (MODELED, transmit-side), residual 1.39× | the 9.6× is a pixel/transmit proxy (modeled:true), not compute; residual "render-the-delta" is a modest real 1.39× |
| **R1 Cycles 3D** | real path-traced scene | **FAIL** | **0.4–0.8× (SLOWER)** at 0.99 | **the critical finding** — on a small/fast render the OIDN denoise + fixed launch overhead EXCEED the sample savings, so low-spp+denoise is *slower* than just rendering high-spp. The synthetic C1/C2 model missed this because it costed purely ∝spp (linear), ignoring fixed/denoise overhead. |

**What real content changed:**
1. **Video transcode/editing is the strongest, realest win (2.18×)** — and it's a core ComputeExchange
   job class ("render/re-encode a long video you don't have the machine for"). This is where to aim first.
2. **Frame-interpolation is real but modest on real footage (~1.3×)**, not the 2.8× synthetic implied —
   real motion has far less speculatable structure.
3. **3D render speculation DID NOT PAY on a small scene (measured 0.4–0.8×)** — a genuine, load-bearing
   correction the synthetic run hid. Low-spp+denoise only wins when the sample savings dominate the
   fixed denoise/launch overhead, i.e. on **large, slow, high-spp production renders** (1080p/4K, complex
   scenes, minutes/frame) — which this 256×256 test was far too small to reach. The next render test must
   use a heavy scene at production resolution; only then does the render regime where speculation pays
   actually show up. Until then, the honest claim is: render speculation is UNPROVEN for us, and negative
   on small scenes.

**Net, honestly:** the methodology is real but **modality-dependent**. It's a clear win for video
transcode, a modest win for frame-interp, transmit-side for images, and **not yet shown for 3D render**
(negative on small scenes; needs a production-scale render to test its real regime). That is a much
truer — and more useful — map than the uniform 2–2.8× the synthetic run suggested. Total session GPU
spend to learn all of this: ~$1.04.

## Track X — CUSTOM 3D-render speculation (our own IP), and the load-bearing regime finding

Built + ran four framework-independent 3D speculation strategies (Blender only as the pluggable
ray-tracer; every WHERE/HOW-much decision is our code) on real GPU renders:

| strategy | net_speedup | quality | reading |
|---|---|---|---|
| X1 CX-Foveate (saliency two-pass) | 0.49-0.73x | 0.56 (bad) | two-pass overhead > one render; composite quality poor |
| X2 spatial tiles (our hardness classifier) | 0.23-0.35x | 0.98 (great) | routing preserves quality, but per-tile Blender launches dominate a fast render |
| X3 bounce-hybrid (direct draft + GI where needed) | 0.82-0.98x | 0.996-1.0 (perfect) | the promising one: near break-even, near-zero quality loss even on a trivial scene |
| X4 temporal reuse (reproject + patch) | - | - | errored on an EXR-reader dependency (not a strategy failure); untested |

THE load-bearing finding (consistent across ALL render experiments C -> X): on an L40S with OptiX
these scenes render in 1-7 SECONDS -- so fast that NO speculation can win, because the fixed overhead
(Blender launch, denoise, multi-pass, composite) dwarfs any compute saved. Render speculation is
REGIME-DEPENDENT: it only pays when the base render is genuinely SLOW (minutes+) -- production 4K /
complex geometry / heavy global illumination / long animations, i.e. exactly the "render something you
don't have the machine for" case. Our deliberately budget-bounded tests (kept < ~90s to save money)
CANNOT reach that regime, so they show a FALSE NEGATIVE. The strategies are sound (X3 preserves quality
perfectly and is architecturally right); they just need an expensive-enough render for smarter/less
compute to save real minutes.

STRATEGIC CONSEQUENCE: the proven near-term win is VIDEO TRANSCODE (2.18x, real). 3D render speculation
is unproven-because-untestable-cheaply -- the decisive experiment is one deliberately EXPENSIVE render
(a GI-dominated or high-res scene whose full render takes minutes, where X3's cheap direct-light draft
is a fraction of the full-GI cost). That costs real GPU-minutes, the one place "keep it cheap" and
"prove render speculation" genuinely trade off. Total session GPU spend through all of this: ~$1.10.

## The DECISIVE render experiments — and the breakthrough (X4 temporal, X5 heavy)

Ran the two decisive tests: X5 a GI-dominated Cornell box (make GI genuinely expensive) and X4
temporal reuse (EXR bug fixed). This settles 3D render speculation:

**X5 bounce-hybrid — DOES NOT pay, and now we know WHY.** On the GI-dominated Cornell box, full GI
(24-32 bounces) was only **1.7× the cost** of the 1-bounce draft (draft 10.9s vs ref 18.7s) — the
honesty guard's threshold was 2×, so it correctly flags this INCONCLUSIVE-toward-negative. The
finding: **bounce DEPTH is not where GPU render cost lives** — the cost is per-sample ray work +
resolution + the first bounce, and deep GI is a minority. So a "cheap direct-light draft" can't save
enough. net_speedup 0.63. Bounce-reduction render-spec is a **dead end** for renders that finish in
seconds-to-tens-of-seconds; its paying regime needs genuinely massive renders (minutes+, 4K, millions
of polys) that cost real GPU-time to even test.

**X4 temporal reuse — the BREAKTHROUGH. 3.7×–12.95× at SSIM ~0.84, 98% of pixels reprojected.**
Rendering ONE keyframe fully, then reprojecting subsequent animation frames by Cycles motion vectors
and re-rendering only disocclusions, gives a **massive** speedup on animation: 3.7× at keyframe-every-8,
**12.95× at keyframe-every-16**, with 98.4% reprojection acceptance. Quality is 0.84 SSIM — a real,
visible tradeoff (reprojection artifacts), and exactly the tunable speed/quality knob: shorter keyframe
intervals raise quality and lower speedup. This is THE 3D render-speculation win, and it is precisely
the product use case — **"render a long animation/video you don't have the machine for."** 0.84@10× is
a great *preview/draft* tier; full render is the *final* tier — a natural tiered-quality product.
