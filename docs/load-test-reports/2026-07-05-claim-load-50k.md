## What was measured

Per the rung's own text: *"Run and commit the currently-missing measurement:
`bench-local` at a 50k-task queue depth with a 100-simulated-poller Go load
generator, plus a `cx_claim_duration` histogram capturing real p50/p90/p99 claim
latency under that load."*

Two things landed to make this possible, both real code (not test-only shims):

1. **`cx_claim_duration_ms` histogram** — `control/scheduler.go` (`claimHistogram`,
   wired into `Store.ClaimTask` via `defer`) + `control/metrics.go` (exposition).
   An in-process, lock-free, atomic-bucket histogram over **every** real
   `ClaimTask` call — the exact `/v1/worker/poll` hot path — bucketed at
   `{1,2,5,10,25,50,100,250,500,1000,2500,5000}` ms. It runs in production today,
   not just under this load test; `/metrics` now exposes
   `cx_claim_duration_ms_bucket/_sum/_count` alongside the existing
   `cx_task_duration_ms` histogram.
2. **A new Go load generator** (`.artifacts/bench-claim-load/loadgen/main.go`,
   not committed — throwaway harness code, kept in `.artifacts/` like
   `bench-local.sh`'s own output — see "Reproducing this" below for the exact
   design) driving **100 concurrent goroutines, each a distinct real registered
   worker identity** (100 real `workers` + `worker_tokens` rows across 10
   suppliers), against a **real 50,000-row `tasks` queue** under one synthetic
   job, hitting the real `GET /v1/worker/poll` HTTP endpoint on a real running
   `cx-control-plane` binary. Each successful claim is immediately released via
   the real `POST /v1/worker/task/{id}/fail` endpoint (`class: transient_io`,
   retryable — requeues in the real 5s `immediateFailBackoff`), so the fleet
   cycles through the backlog for the whole measurement window instead of each
   worker holding one task forever.

## Headline result

| scenario | pollers | queue depth | wall window | claims | claims/sec | claimed-request p50 | p90 | p99 |
|---|---|---|---|---|---|---|---|---|
| light | 10 | 50,000 | 60s (fixed) | 100 | **1.7/s** | 2,572 ms | 2,644 ms | 2,698 ms |
| heavy | 100 | 50,000 | 90s target (ran 107s; client-side 30s timeouts hit) | 209 | **1.95/s** | 23,678 ms | 25,068 ms | 29,798 ms (many timed out at 30s) |

**This replaces the doc's estimated ceiling — and the real number is worse than
the estimate.** The parent section's text read: *"the claim query's cost grows
with queue depth via correlated subqueries evaluated before the final
`ORDER BY LIMIT 1` (estimated collapse around 10-50 claims/sec at a 10k+
backlog)."* The measured throughput at 50k depth is **1.7–2.0 claims/sec fleet-wide**
— below the low end of that estimate — and it does **not** improve when going from
10 to 100 concurrent pollers (more pollers just means more requests queued up
behind the same per-claim cost, confirmed by the 191/400 = 48% client-timeout rate
at 100 workers).

## Root cause, measured directly (not inferred)

This is the useful part beyond "it's slow" — the exact mechanism, isolated with
`EXPLAIN (ANALYZE, BUFFERS)` on the literal production query
(`control print-claim-sql`, piped through `PREPARE`/`EXECUTE` with real bind
values, since that's what pgx actually sends — a plain-text `psql` run of the
same SQL with literal constants inlined is **not** equivalent and is
misleadingly fast):

- The **pinned branch** (`t.claimed_by = $1 AND t.started_at IS NULL` — the rare
  tiebreak/hedge-dispatch path) is fast regardless of queue depth: **~14 ms**,
  correctly using `tasks_pkey`/`workers_supplier_idx`. Not the bottleneck.
- The **general branch** (`t.claimed_by IS NULL` — the common path essentially
  every real dispatch takes) is genuinely **bimodal** at 50k depth:
  - **Fast mode (~11–15 ms):** the ~50k-row candidate set's `ORDER BY` sort (6
    computed tiebreak columns wide, per the routing/warm/fairness preference
    terms) fits in `work_mem` (4 MB default) as an in-memory quicksort.
  - **Slow mode (~1.85–2.7 s per claim, isolated single-request — climbing to
    20–30 s under 100-way concurrency):** the same sort **spills to disk** —
    `EXPLAIN` reported `Sort Method: external merge Disk: 4800kB` — because the
    sorted row set (candidate rows × the routing/warm/fairness tiebreak columns)
    exceeds `work_mem`. Under genuine concurrency, many sessions compete for
    the same bounded `work_mem` and OS page cache simultaneously, so the slow
    mode dominates — this is exactly what the 100-poller run shows: not a
    modest per-request slowdown, but **request latency climbing from ~2.5 s to
    20–30 s** as concurrent claims contend for sort memory and I/O.
  - Both modes plan the SAME query shape (confirmed via literal repeat
    `EXPLAIN ANALYZE EXECUTE` against an unchanged 50k-row queue) — this is a
    **resource-contention-triggered plan cost**, not a one-off bad plan choice.
- `tasks_ready_unclaimed_idx` (the partial index this whole section is named
  after) **is used** correctly in every plan observed (`Bitmap Index Scan using
  tasks_ready_unclaimed_idx`) — it is not the limiting factor at this depth. The
  bottleneck is entirely downstream: sorting the ~50k eligible candidates by
  the six-term routing/tiebreak `ORDER BY` before `LIMIT 1` can extract the
  winner.

This directly confirms — with real numbers, not a read-the-diff guess — the
doc's own diagnosis for this facet ("the claim query's cost grows with queue
depth via correlated subqueries evaluated before the final `ORDER BY LIMIT 1`"),
and additionally narrows the mechanism for whoever picks up rung 6→7 ("Make the
real claim query cheap regardless of queue depth"): the fix needs to either (a)
push the six-term tiebreak ranking to only run over a small, already-narrowed
candidate set (e.g. `LIMIT`-then-rank instead of rank-then-`LIMIT`), or (b) cut
the columns actually being sorted, or (c) raise `work_mem` for this connection
role — (c) alone is a stopgap since cost still grows with queue depth, just with
a higher spill threshold.

`cx_claim_duration_ms_sum` = 293,813 ms over 3,115 calls (mean 94 ms, pulled up
from a ~13 ms 204-path mode by the ~100 slow 200-path claims — consistent with
the bimodal mechanism above). Full scrape:
`.artifacts/bench-claim-load/evidence/metrics_scrape_10worker_run.txt`.

100-worker run (histogram scraped immediately after the run, before the process
was restarted for the 10-worker run — transcribed to
`.artifacts/bench-claim-load/evidence/metrics_scrape_100worker_run.txt` since the
live scrape wasn't piped to disk before restart, see that file's neighboring
`README.txt`): **all 400 observed calls landed in the `+Inf` bucket** (every
single claim during the 100-worker run took longer than the histogram's own
5000 ms top bucket bound) — `cx_claim_duration_ms_sum` = 10,591,091 ms / 400 =
**26.5 s mean**. This is itself a finding: the histogram's bucket boundaries
were sized for the doc's original (healthier) expectation of the claim path, and
the real pathological-load numbers blew straight through them. A follow-on
pass should either widen the top bucket or fix the underlying query — the
histogram code itself (bucket boundaries in `claimDurationBucketsMs`,
`control/scheduler.go`) is a one-line change either way.

## Queue and DB state, confirmed via direct SQL (not inferred from HTTP responses)

- 50,000 real `tasks` rows inserted under one synthetic `jobs` row
  (`job_type='embed'`, `tier='batch'`, uncapped `max_usd`), matching the
  columns and shape a real job submission populates (including `result_key`,
  required for the real presign step in `handleWorkerPoll` to succeed — an
  earlier iteration of this run omitted it and got a 500; fixed before the
  measured runs above).
- 100 real `workers` + `worker_tokens` rows across 10 real `suppliers`
  (`hw_class='apple_silicon_max'`, broad `supported_jobs`/`supported_models`,
  `min_payout_usd_hr=0`, `thermal_ok=true`) — genuinely eligible under
  `ClaimTask`'s real hard filter, not a synthetic bypass.
- Postgres backend CPU sampled mid-run via `ps aux` during the 100-worker test:
  individual `postgres: cx cx ...` backends at **13–21% CPU each**, ~100
  concurrently active — genuine CPU saturation on the 12-core test machine, not
  lock contention (`pg_stat_activity` showed `state=active` with no
  `wait_event` on the stalled backends, i.e. real work, not blocking).
- Final state after the 100-worker run: 209 tasks cycled through claim→fail→
  retrying (`task_failures` rows = 209, matching `loadgen`'s `claimed` count),
  49,791 left queued, 0 stuck in `running` — the queue never got a chance to
  meaningfully drain at ~2 claims/sec against a 50k backlog.

## Honest caveats

- **Single-box test, not the prod topology.** Postgres, MinIO, and the
  control-plane binary all ran on one Apple M3 Pro laptop, competing for the
  same 12 cores — not `docker-compose.prod.yml`'s separate containers on a
  cloud droplet with its own resource caps (`mem_limit`/`cpus` per service).
  Absolute numbers will differ on the real droplet; the **mechanism** (sort
  spill under concurrent claim load at deep queue depth) is a property of the
  query and Postgres's planner, not this specific machine, so it should
  reproduce qualitatively anywhere `work_mem` is anywhere near its 4 MB
  default under this row width.
- **The 100-worker run did not cleanly finish inside its 90s target** — client
  requests use a 30s HTTP timeout (a reasonable real-world worker timeout), and
  48% of requests hit it. The reported percentiles for that run are therefore a
  mix of genuine completions and requests cut off by the client timeout; the
  204/200 fleet-wide throughput number (1.95 claims/sec) is the reliable
  headline figure from that run, not the percentile tail.
- **`cx_claim_duration_ms`'s bucket boundaries are undersized** for this
  pathological case (see above) — the histogram correctly proves "everything
  was slow" but can't resolve p50/p90/p99 *within* the slow regime from the
  Prometheus buckets alone; the load generator's own wall-clock percentiles
  (reported in the headline table) fill that gap for this report, cross-checked
  against the histogram's `_sum`/`_count` (both agree on the ~1.85s-per-call
  single-request baseline).
- **This is a snapshot, not a regression suite.** Nothing here is wired into CI
  or `bench-local.sh`; a future change to the claim query should be re-measured
  the same way (harness described below) to confirm it actually moved these
  numbers, not just that it "looks right" in `EXPLAIN`.

## Reproducing this

Infra (throwaway, matching `docker-compose.prod.yml`'s shape):
```
initdb + pg_ctl -D pgdata -o "-p 5459 -c unix_socket_directories=/tmp/cx_bundleSC_pgsock" ...
minio server minio-data --address localhost:9169 --console-address localhost:9170
mc mb cxbench/cx-jobs
psql ... -f db/schema.sql
```
Seed 100 workers/tokens across 10 suppliers + 50,000 queued tasks under one job
(SQL committed alongside this report is NOT included — it was throwaway,
see the shape described above and in `db/schema.sql`'s `tasks`/`workers`/
`worker_tokens` tables; regenerate with a loop over `INSERT INTO workers/
worker_tokens/tasks` matching `control/seed.go`'s seeded-worker shape).

Load generator: 100 (or 10) goroutines, each with a distinct `X-Worker-Token`,
looping `GET /v1/worker/poll` → on `200`, immediately
`POST /v1/worker/task/{id}/fail {"class":"transient_io"}` to release the claim
and cycle → on `204`, brief backoff. Fixed-duration window (not drain-to-zero,
since a real fleet runs against a standing backlog, not a race to empty it).
Wall-clock latency recorded client-side; cross-checked against the server's own
`cx_claim_duration_ms` histogram scraped from `/metrics` at the end of the run.
