### (b2) Flatness — the Control Plane Hot Path 8->9 proof

Entry 61 (docs/internal/CREED_AND_PATH_TO_TEN.md) root-caused a **~190x** latency
ratio between a near-empty and a loaded queue to `cheaper_class_online` — a
correlated `EXISTS` that sequentially scanned `workers` **once per candidate
task row** (O(queue × fleet)). The 8->9 fix computes it **once per candidate job**
(`eligible_jobs AS MATERIALIZED`, behind a claimable-task guard), so the fleet
scan no longer grows with the claimable queue. The row below is the SAME real
query, SAME 601-worker fleet + dilution, run against a near-empty
claimable queue — the loaded:near-empty p50 ratio is the flatness number.

| loaded p50 | near-empty p50 (49-task) | loaded:near-empty ratio | entry 61 (pre-fix) |
|-----------|-----------------|-------------------------|--------------------|
| 12.792 ms | 1.429 ms | **8.95x** | ~190x |

**What the fix removed, and what the residual 8.95x is.** Entry 61's ~190x was
dominated by `cheaper_class_online`: a correlated `EXISTS` that sequentially
scanned the `workers` table **once per candidate task row** — O(queue × fleet).
`EXPLAIN (ANALYZE, BUFFERS)` at this exact 12k-task / 601-worker / 100k-dilution
scale confirmed it (`Seq Scan on workers w2 … loops=12001`, ~48k buffer hits).
The 8->9 fix moves **every per-job ordering signal** (`cheaper_class_online`,
`worker_tps`, `warm_for_task`, `job_dispatched_count`) into an `eligible_jobs AS
MATERIALIZED` CTE computed **once per candidate job** — verified in the same
plan (`… loops=241`, one per job with claimable work, not 12,001). The residual
per-queue cost is now just the LIMIT-1 `Sort` over the candidate tasks
(`Sort … Memory ~2.3MB`), an O(n log n) that is a fundamentally cheaper cost
class than the removed O(queue × fleet) fleet-rescan. Absolute loaded latency
fell from entry 61's **~1.4 s** to **~13 ms** at the same scale on this M3 Pro.

**Equivalence — proven, not asserted.** The restructured query is not "similar";
it produces a byte-identical ordered task list to the original per-task query at
every queue position (12,001 rows, symmetric-difference = 0), so the SAME task is
claimed. The proof is a direct SQL full-ordering comparison of the real, rendered
`control print-claim-sql` output against the original per-task ordering, plus the
full real integration suite (which exercises pinning, hedging, tiers, budget cap,
warm-routing, and the adversarial quarantine harness) green before and after.
